# 身份卡数据

> 更新日期：2026-07-22
> 对应版本：`0.4.13-internal`

身份卡用于决定玩家开局的现金流起点。当前版本将身份卡简化为“职业角色卡”，只保留会直接影响游戏结果的字段。

## 字段约定

| 字段 | 含义 | 是否展示 | 是否影响游戏 |
|---|---|---|---|
| `id` | 身份卡唯一标识 | 否 | 是，用于选择和开始游戏 |
| `name` | 职业角色名 | 是 | 是，作为玩家身份名称 |
| `income` | 常规月收入 | 是 | 是 |
| `expense` | 常规月支出 | 是 | 是 |
| `savings` | 初始现金储备 | 是 | 是 |

当前不使用这些字段：

| 字段 | 移除原因 |
|---|---|
| `age` | 暂不影响游戏机制 |
| `job` | 和 `name` 重复，统一用职业角色名表达 |
| `family` | 暂不影响游戏机制 |
| `description` | 暂不影响游戏机制，且容易变成说明文字堆叠 |

身份页不展示初始安全垫，避免玩家只为了抽到更高安全垫而反复重抽。安全垫仍会在正式游戏过程中展示和结算。

## 职业专属事件

当前 14 个预设身份均配置了专属职业事件。新局会在第 2 至第 5 个月之间隐藏安排一次；职业不匹配时，对应卡不会进入正式卡池。

| 身份 | 职业事件 | 主要选择 |
|---|---|---|
| 企业白领 | 跨部门项目邀请 | 接下项目 / 保持原工作节奏 |
| 自媒体博主 | 平台商务合作 | 完整合作 / 轻量合作 / 不接 |
| 餐饮老板 | 后厨设备升级 | 更换设备 / 继续维修 |
| 咖啡主理人 | 商圈客流变少 | 全天营业 / 缩短营业 |
| 钢琴老师 | 成人钢琴小班 | 接下小班 / 保持一对一教学 |
| 销售 | 大客户订单 | 完整订单 / 部分订单 |
| 高级工程师 | 核心系统升级 | 负责改造 / 只做技术支持 |
| 数据分析师 | 临时数据项目 | 购买分析工具 / 沿用现有工具 |
| 建筑师 | 方案竞标 | 独立竞标 / 联合竞标 |
| 医生 | 进修名额 | 自费进修 / 增加门诊 |
| 运动员 | 赛季装备更新 | 更换装备 / 继续使用 |
| 程序员 | 线上故障值班 | 独立值班 / 同事轮值 |
| 收纳师 | 全屋收纳加单 | 独立完成 / 找助手 / 只做核心区域 |
| 图书管理员 | 周末阅读活动 | 自己主持 / 协调外部讲者 |

自媒体博主的“平台商务合作”按当前角色月收入计算：完整合作带来月收入 40% 的一次性收入，轻量合作带来月收入 20% 的一次性收入；不改变常规月收入基数。

咖啡主理人另有“邀请探店博主”和“门口来了一只流浪猫”两张职业限定随机事件。自定义身份不会匹配预设职业事件。

## 调整原则

- 职业名尽量清楚、生活化，避免同时出现“身份”和“职位”两套说法。
- 每张身份卡的差异应主要体现在收入、支出、现金储备，而不是描述文案。
- 月收入、月支出、现金储备需要贴近用户直觉，不要为了难度刻意极端化。
- 可以保留不同压力区间，便于测试不同局面：
  - 低收入低支出：如图书管理员、钢琴老师。
  - 中等收入中等支出：如企业白领、数据分析师、销售。
  - 高收入高支出：如医生、餐饮老板、运动员。
  - 高收入较稳储备：如高级工程师、程序员。
- 如果未来要加入年龄、家庭、行业风险等信息，建议先把它们转化成明确机制，例如固定支出、收入波动、事件权重或专属事件池，再放回身份卡。

## 数据维护位置

当前实际数据在 [identity-cards.js](./identity-cards.js) 的 `identityCards` 数组中，游戏和数值模拟共用这份数据。

如果修改本文件中的表格，也需要同步修改 `identity-cards.js`，否则游戏不会读取到新数据。修改后运行 `npm test` 和 `npm run simulate`，检查字段与数值基线。
