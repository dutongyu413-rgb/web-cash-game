# 现金流人生地图

一个手机端现金流投教小游戏。玩家抽取身份，按月份掷骰前进，经历收入、支出、健康、选择和长期计划事件，观察现金储备和安全垫如何变化。

## 当前状态

- 核心玩法机制基本可测：身份、掷骰、地图、事件卡、本月结算、历史记录、最终结算。
- 事件卡已拆到 `event-cards.js`，便于单独调整文案、金额、持续时间和选项。
- 身份卡已调整为职业角色卡，只展示职业、现金储备、月收入、月支出，不在开局阶段展示安全垫。
- 当前定投机制已增加“估值修复”和“阶段高估”两个中途市场事件，玩家可以在这些事件中卖出一部分或全部持仓。
- 页面样式仍在迭代中，当前更适合内部试玩和收集反馈。

## 如何运行

最简单方式：

1. 直接打开 `index.html`。
2. 或启动本地服务：

```bash
npm run dev
```

默认访问地址见终端输出，一般是 `http://127.0.0.1:4173`。

## 文件结构

```text
index.html          页面入口，按顺序加载样式、事件卡和主脚本
style.css           全部界面样式
event-cards.js      事件卡牌数据

dev-server.mjs      简单本地静态服务

```

## 身份卡怎么改

身份卡当前仍在 `script.js` 的 `identityCards` 中维护，详细数据单独整理在 [IDENTITY_CARDS.md](./IDENTITY_CARDS.md)。

当前身份卡只保留会直接影响游戏的字段：

- `id`：唯一标识，不要重复。
- `name`：职业角色名，也是页面唯一显示的身份名称。
- `income`：常规月收入。
- `expense`：常规月支出。
- `savings`：初始现金储备。

身份卡不再维护 `age`、`job`、`family`、`description`。这些字段如果不转化为收入、支出、现金储备或事件机制，就不会影响游戏体验，暂时不放进身份卡。

身份页也不展示初始安全垫，避免玩家为了追求更高安全垫反复抽卡。安全垫仍然是游戏内核心指标，会在地图页、月度结算和最终结算中展示。

## 事件卡怎么改

事件卡集中在 `event-cards.js`。每张卡大致长这样：

```js
{
  id: "kids_theme_park",
  title: "带孩子去游乐场",
  category: "one_time_cost",
  description: "周末带孩子去游乐场，门票、交通和餐饮一次性支出 1200 元。",
  effect: { type: "change_savings", amount: -1200 },
  insight: "生活里的快乐也会花钱，但它不是财务错误，只是需要被看见。",
}
```

常用字段：

- `id`：唯一标识，不要重复。已经上线试玩后尽量不要随便改旧卡 `id`。
- `title`：卡牌标题。
- `category`：卡牌类型，会影响抽卡池和卡牌颜色。
- `description`：玩家看到的事件描述。
- `effect`：财务影响。
- `choices`：可选项卡牌使用，替代 `effect`。
- `insight`：本月结算里的简短分析。
- `group`：可选，用于控制同组卡牌频率，例如兴趣类事件。

当前 `category`：

- `expense_up`：持续支出增加
- `one_time_cost`：一次性支出
- `income_down`：收入下降或不确定
- `positive`：正向补给
- `health_risk`：健康风险
- `choice`：选择事件

## 当前支持的 effect 类型

- `change_savings`：直接改变现金储备。
- `change_savings_by_income_percent`：按月收入比例改变现金储备。
- `one_month_income_change`：只影响本月收入，固定金额。
- `one_month_income_percent`：只影响本月收入，按比例。
- `one_month_expense_change`：只影响本月支出，固定金额。
- `one_month_expense_percent`：只影响本月支出，按比例。
- `add_active_effect`：增加持续影响，可影响收入、支出或比例。
- `add_uncertain_active_effect`：增加不确定持续影响，例如收入下降但恢复时间未知。
- `schedule_active_effect`：若干月后生效的持续影响。
- `start_dca_plan`：开启定投计划。
- `start_protection_plan`：开启基础保障。
- `career_course_plan`：职业课程，后续 3-5 个月可能触发收入提升。
- `buy_car`：换车的复合逻辑。
- `compound`：组合多个 effect。
- `none`：无财务影响。

如果要新增 effect 类型，需要在 `script.js` 里同步处理这些位置：

- `applyEffect`
- `effectText`
- `estimateStress`
- 如果影响结算展示，也要看 `buildMonthlySummary` 和 `showMonthlySummary`

## Umami 访问和行为追踪

当前项目已预留 Umami 统计接入，用于长期链接发布后观察访问量和关键行为。

### 如何启用

在 [index.html](./index.html) 中找到：

```html
window.CASH_GAME_UMAMI_WEBSITE_ID = "";
```

把空字符串替换为 Umami 后台提供的 `data-website-id`：

```html
window.CASH_GAME_UMAMI_WEBSITE_ID = "你的 Umami Website ID";
```

留空时不会加载 Umami，也不会发送任何统计数据。

### 项目区分方式

如果多个 GitHub Pages 项目共用同一个域名，例如：

```text
dutongyu413-rgb.github.io
```

本项目会给所有自定义事件自动加上：

```js
project: "cash_game"
```

同时事件名统一使用 `cash_game_` 前缀，便于和其他项目区分。

### 隐私边界

统计事件不会上传这些具体数据：

- 月收入
- 月支出
- 现金储备
- 安全垫具体数值
- 具体生存分数
- 自定义身份中输入的金额

会上传的是非敏感分档和行为信息，例如：

- 项目标识：`cash_game`
- 挑战长度：12、24 或 36
- 事件类型：健康、收入、选择等
- 是否发生卖出
- 安全垫区间，例如 `1_to_3_months`

## 当前定投和卖出逻辑

当前版本暂时没有完整的基金净值线，但已经在原有定投机制上补了一条轻量后续线。

### 触发流程

1. 玩家抽到“低估区域的定投机会”。
2. 选择“开始定投”后，每月增加 2000 元定投支出。
3. 后续会触发“估值修复”。
4. 如果玩家仍有持仓，后续可能触发“阶段高估”。
5. 玩家可以在后续事件中卖出一半或全部卖出。

### 卡牌文案原则

投资后续卡牌只描述市场状态，不替玩家判断账户状态，也不写引导性分析。

当前保留的市场文案：

- `估值修复`：市场从低估区域回到正常估值附近。
- `阶段高估`：市场进入阶段性高估区域。

不要在卡牌里写这些内容：

- “你的账户已经接近回本。”
- “你的账户出现明显浮盈。”
- “现在卖出可以……”
- “继续持有可能……”

玩家自己的状态用数字面板展示：

- 累计投入
- 当前估算市值
- 阶段收益率

### 选择项逻辑

如果定投仍在进行：

- `继续定投`：保持每月投入和当前持仓。
- `停止定投`：停止后续投入，保留已有持仓。
- `卖出一半`：卖出约 50% 持仓，回收现金。
- `全部卖出`：卖出全部持仓，回收现金。

如果定投已经暂停：

- `继续持有`：保持当前持仓。

### 卖出计算

卖出一半和全部卖出使用同一套计算逻辑。

```text
卖出本金 = 当前持仓本金 × 卖出比例
卖出金额 = 卖出本金 × (1 + 当前阶段收益率)
现金储备 += 卖出金额
当前持仓本金 -= 卖出本金
累计卖出本金 += 卖出本金
累计卖出金额 += 卖出金额
已实现盈利/亏损 += 卖出金额 - 卖出本金
```

只要玩家选择卖出，后续定投就会停止：

- 卖出一半：状态变为已暂停，剩余持仓继续保留。
- 全部卖出：状态变为已卖出，剩余持仓归零。

### 最终结算展示

最终结算页需要区分“已经到手的钱”和“还没卖出的估算”。

如果发生过卖出：

- 显示已实现盈利或已实现亏损。
- 计算方式：`卖出金额 - 卖出本金`。

如果还有剩余持仓：

- 显示期末持仓估算。
- 显示未实现盈利或未实现亏损。
- 计算方式：`期末持仓估算 - 剩余持仓本金`。

如果全部卖出：

- 只显示已实现盈利或已实现亏损。
- 不显示未实现部分。

## 试玩重点

建议同事测试时重点看这些问题：

- 身份卡的职业、收入、支出、现金储备是否符合直觉。
- 卡牌金额是否贴近生活。
- 同类卡牌是否出现过于频繁。
- 按比例变化的事件是否比固定金额更合理。
- 保障生效时，玩家是否能明显感知到“少花了钱”。
- 定投后续的估值修复、阶段高估和卖出选择是否容易理解；卖出后现金储备、已实现盈利和未实现盈利是否正确展示。
- 地图移动、抽卡、本月结算是否有游戏感。
- 最终结算是否像一局游戏的战报，而不是财务报表。
- 手机窄屏下是否有文字拥挤、按钮换行、内容重叠。

## 手动测试流程

建议至少跑这几类局：

1. 随机身份，完整玩到 12 个月。
2. 选择 36 个月，主动结束游戏，查看最终结算。
3. 配置基础保障后，尽量触发健康风险卡，观察保障提示和结算。
4. 开启定投后玩到结束，观察估值修复、阶段高估、卖出选择和最终定投阶段结论。
5. 分别测试“卖出一半”和“全部卖出”，确认现金储备增加、后续定投停止，最终页能区分已实现盈利和未实现盈利。
6. 选择“离公司近一点”，确认 3 个月后收入提升是否生效。
7. 用支出压力较高的身份测试现金储备被快速消耗的压力局。
8. 用图书管理员、教师等低收入或稳态收入身份测试低收入局。

## 已知待优化方向

- 页面视觉还需要继续打磨，尤其是地图和事件卡的游戏感。
- 本月结算可以继续减少“报表感”，强化回合结果。
- 最终结算可以继续做成更完整的战报。
- 身份页已经改成角色卡方向，后续可继续优化卡牌质感和数值展示。
- 卡牌池还可以继续扩充，减少重复抽到同类事件的感觉。
- 投资玩法当前只做轻量的估值修复和阶段高估事件，后续如需升级为完整基金净值线，可在本地维护文档中继续设计。

## 注意

本项目用于现金流管理和投资者教育场景下的模拟体验，不构成投资建议或收益承诺。游戏中的收入、支出、市场表现和事件结果均为简化模拟。
