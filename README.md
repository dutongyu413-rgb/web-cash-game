# 现金流人生地图

一个手机端现金流投教小游戏。玩家抽取身份，按月份掷骰前进，经历收入、支出、健康、选择和长期计划事件，观察现金储备和安全垫如何变化。

## 当前状态

- 核心玩法机制基本可测：身份、掷骰、地图、事件卡、本月结算、历史记录、最终结算。
- 事件卡已拆到 `event-cards.js`，便于单独调整文案、金额、持续时间和选项。
- 事件卡当前共 71 张，包含普通随机事件、选择事件、后续事件、职业限定机会和 13 张职业专属事件。
- 身份卡已调整为职业角色卡，只展示职业、现金储备、月收入、月支出，不在开局阶段展示安全垫。
- 当前定投机制已增加“估值修复”和“阶段高估”两个中途市场事件，玩家可以在这些事件中卖出一部分或全部持仓。
- 当前卡牌已增加轻量后续事件：换车、旅行推迟、兴趣课程、户外徒步、瑜伽课等选择会在后续月份产生回响。
- 地图下方只展示正在生效的长期计划和临时支出；买车、推迟旅行等后续事件默认不展示“待生效”，保留随机感。
- 最终结果页“最重一击”已改为“累计冲击”口径，按玩家实际经历过的收入减少、支出增加和现金储备消耗累计，不再按最坏情况预估。
- 月度结算、历史记录和结果轨迹现已共用月度快照；历史金额记录结算后的实际储备，不再记录事件刚生效但尚未结算的中间值。
- 主动结束改为二次确认，并按实际完成月份生成“阶段结果”；最终月的后续事件会全部展示后再进入结果页。
- 已加入可复现随机种子和 Node 核心规则测试，便于内部测试复现卡牌与财务计算问题。
- 第二轮新手体验已完成：首次开局显示三步引导，之后不再自动打断；本局菜单可重新查看玩法说明。
- 地图页已收敛为月份、现金储备、安全垫、月收入和月支出五项核心信息；结束和重开入口移入菜单。
- 事件增加短暂揭晓动效和选项确认反馈；本月结算只保留本月现金流、现金储备和实际特殊影响，不再展示“本月分析”。
- 第三轮内部测试能力已完成：调试模式增加可视化 TEST 面板，可固定种子、指定事件、修改月份和储备、重置引导并复制诊断信息。
- 自动测试现已覆盖核心财务规则、71张事件卡字段与效果、13个身份字段、职业事件隔离，以及数值模拟的可复现性。
- 已加入批量数值模拟器，当前基线共运行14,040局，只用于发现异常，不干预正式游戏的随机抽卡。
- 页面样式仍在迭代中，当前更适合内部试玩和收集反馈。
- 挑战长度保留12、24、36个月，默认选择24个月；所有模式仍按月正常推进。
- 普通事件不再要求点击“确认影响”；系统直接完成月度结算，并显示居中的结果卡。结果卡不会自动消失，点击卡片外遮罩后关闭。
- 选择事件窗口改为屏幕居中；完成选择后进入同一套结果卡。基础保障减损在结果卡中单独强调，现金流击穿仍保留完整结算。
- 首页海报增加480、768、941三档WebP响应式资源；地图背景和棋子增加WebP版本与PNG回退，并在进入游戏前预加载。
- 本地静态服务为图片增加ETag和短期缓存，重复访问时减少资源传输。
- 定投机会不再在第10个月强制出现；估值修复和阶段高估按照36个月自然节奏安排，不会压缩到短局结束前。
- 第一批普通事件已改为选择卡：房租上涨、车辆维修、家电故障、项目延期回款、闲置物品卖出和牙齿治疗均提供不同现金流取舍。
- 家电维修、车辆简修、延期尾款和闲置物品挂售会按真实月份触发隐藏后续；短局结束前未到期时不会提前补发。
- 13个预设身份各有1张职业专属事件；每局第2至第5个月强制触发当前身份对应卡，其他职业卡不进入随机池。
- 普通职业事件增加语义资格和软权重：明显不符合工作方式的卡牌会被排除，其余相关事件只轻微调整概率；运动员不会再抽到客户预算、提成或公司福利类事件。
- 运动员选择继续使用磨损装备后，三个月后按概率触发结果；发生训练意外时会同时减少本月收入和现金储备。
- 运动员增加商务活动出席和品牌代言邀约两张限定正向事件；商务活动相对常见，品牌代言更少见但收入更高，普通副业与兼职单的概率相应降低。
- “转盘抽事件 + 月份时间线”已作为内部备选方案留档；下一步先做独立原型对比，不直接替换当前地图。
- 最终评分增加“生活体验”维度：明确牺牲休息、心情或生活安排的选择会扣分，避免只靠压缩支出获得满分；普通储蓄和投资不会仅因少花钱被扣分。
- 结果页增加评分说明、提交试玩反馈、用当前身份再玩、换个身份和返回首页；未配置在线问卷时，反馈会通过手机分享或复制文本交付。
- 快速结果与完整结算直接显示同一时点的最终数字，仅保留短暂强调动效，避免滚动数字与本月合计暂时矛盾。

- 结果页可以查看评分结构和本局生活体验来源；100分表示完成度、财务状态和生活状态的综合结果，不代表存在唯一正确答案。
- “提交试玩反馈”已接入飞书问卷，并附带版本、身份、挑战长度、完成月份、结束方式和等级等非敏感参数；不会传递具体财务金额。
- “用当前身份再玩”保留身份与挑战长度，“换个身份”重新抽取身份，“返回首页”回到封面。

### 定投自然周期

- “低估区域的定投机会”只在前10个月随机进入卡池，一局最多出现一次，不做第10个月保底。
- 取消第6个月后的额外直出概率。
- 开始定投后约8-14个月才可能进入估值修复。
- 估值修复后约10-18个月才可能进入阶段高估。
- 触发月份允许晚于当前挑战结束月份；12个月局可能完全没有投资剧情或停留在低估阶段。

### 普通事件选择化

- 房租上涨可在持续涨租与一次性搬家支出之间选择。
- 车辆维修和家电故障增加“先少花、后续再处理”的分支。
- 牙齿治疗可一次支付 20,000 元，或分 12 次每月支付 2,500 元。
- 项目奖金延期时可等待两个月：50% 概率正常收到月收入 60% 的奖金，50% 概率诉讼追回并扣除月收入 15% 的成本；也可立即领取月收入 40% 的折价奖金。
- 闲置物品可以立即卖出获得 1,200 元，也可以分别挂售；两个月后按固定随机种子结算 1,800 元或 600 元。

### 职业专属事件

- 13 张职业卡分别绑定 13 个预设身份，不匹配的职业卡不会进入正式卡池。
- 新局会在第 2 至第 5 个月隐藏安排一次职业事件，触发后不再重复。
- 当前事件为：跨部门项目邀请、平台商务合作、后厨设备升级、公共课程项目、大客户订单、核心系统升级、临时数据项目、方案竞标、进修名额、赛季装备更新、线上故障值班、全屋收纳加单、周末阅读活动。
- 自定义身份没有职业标签，当前不会强制触发职业专属事件。

## 身份卡怎么改

身份卡当前在 `identity-cards.js` 中维护，浏览器游戏和数值模拟读取同一份数据；详细字段说明整理在 [IDENTITY_CARDS.md](./IDENTITY_CARDS.md)。

当前身份卡只保留会直接影响游戏的字段：

- `id`：唯一标识，不要重复。
- `name`：职业角色名，也是页面唯一显示的身份名称。
- `income`：常规月收入。
- `expense`：常规月支出。
- `savings`：初始现金储备。

身份卡不再维护 `age`、`job`、`family`、`description`。这些字段如果不转化为收入、支出、现金储备或事件机制，就不会影响游戏体验，暂时不放进身份卡。

身份页也不展示初始安全垫，避免玩家为了追求更高安全垫反复抽卡。安全垫仍然是游戏内核心指标，会在地图页、月度结算和最终结算中展示。

## 事件卡怎么改

事件卡集中在 `event-cards.js`。每张卡大致长这样：

```js
{
  id: "kids_theme_park",
  title: "带孩子去游乐场",
  category: "one_time_cost",
  description: "周末带孩子去游乐场，门票、交通和餐饮一次性支出 1200 元。",
  effect: { type: "change_savings", amount: -1200 },
  insight: "生活里的快乐也会花钱，但它不是财务错误，只是需要被看见。",
}
```

常用字段：

- `id`：唯一标识，不要重复。已经上线试玩后尽量不要随便改旧卡 `id`。
- `title`：卡牌标题。
- `category`：卡牌类型，会影响抽卡池和卡牌颜色。
- `description`：玩家看到的事件描述。
- `effect`：财务影响。
- `choices`：可选项卡牌使用，替代 `effect`。
- `insight`：保留的分析字段，当前版本不在本月结算和最终报告中展示。
- `group`：可选，用于控制同组卡牌频率，例如兴趣类事件。

当前 `category`：

- `expense_up`：持续支出增加
- `one_time_cost`：一次性支出
- `income_down`：收入下降或不确定
- `positive`：正向补给
- `health_risk`：健康风险
- `choice`：选择事件

## 当前支持的 effect 类型

- `change_savings`：直接改变现金储备。
- `change_savings_by_income_percent`：按月收入比例改变现金储备。
- `one_month_income_change`：只影响本月收入，固定金额。
- `one_month_income_percent`：只影响本月收入，按比例。
- `one_month_expense_change`：只影响本月支出，固定金额。
- `one_month_expense_percent`：只影响本月支出，按比例。
- `add_active_effect`：增加持续影响，可影响收入、支出或比例。
- `add_uncertain_active_effect`：增加不确定持续影响，例如收入下降但恢复时间未知。
- `schedule_active_effect`：若干月后生效的持续影响。
- `schedule_savings_effect`：若干月后改变现金储备，用于轻量后续事件。
- `schedule_savings_by_income_percent`：若干月后按角色基础收入比例增加现金储备，用于延期回款。
- `schedule_random_savings_effect`：若干月后从带权重的结果中抽取一个，改变现金储备。
- `invest_or_reserve`：固定金额到账后，如果已有正在进行的定投则按比例加码定投，否则进入现金储备。
- `bonus_invest_or_reserve`：年终奖专用，金额按 1 个月收入计算；如果已有正在进行的定投则按比例加码定投，否则进入现金储备。
- `start_dca_plan`：开启定投计划。
- `start_protection_plan`：开启基础保障。
- `career_course_plan`：职业课程，后续 3-5 个月可能触发收入提升。
- `buy_car`：换车的复合逻辑。
- `compound`：组合多个 effect。
- `none`：无财务影响。

如果要新增 effect 类型，需要在 `script.js` 里同步处理这些位置：

- `applyEffect`
- `effectText`
- `choiceEffectText` 和 `choiceEffectTone`，如果它会出现在选择按钮中。
- `recordActualMonthStress` / `getActiveEffectMonthlyStress`，如果它会影响最终结果页的“累计冲击”。
- 如果影响结算展示，也要看 `buildMonthlySummary` 和 `showMonthlySummary`

## 正向选择卡当前规则

正向卡不再只做“回血包”。当前已改成选择卡的正向事件包括：

- `year_end_bonus` 年终奖到账：
  - 补进现金储备：现金储备增加 1 个月收入。
  - 加码定投或先存着：如果已有定投计划，一半奖金加码定投，一半进入储备；如果没有定投计划，整笔进入储备。
  - 安排一次家庭犒劳：描述为餐厅和近郊周末，保留 65% 奖金进储备。
- `tax_refund` 退税或补贴到账：
  - 补进现金储备：现金储备 +8000。
  - 加码定投或先存着：如果已有定投计划，一半补贴加码定投，一半进入储备；如果没有定投计划，整笔进入储备。
  - 换一件耐用品：描述为更换床垫，现金储备 +3200。
- `side_income` 副业收入：
  - 接下这单：本月收入增加约 22%，同时本月支出增加 1200，代表交通、工具和外包杂费。
  - 只接小单：本月收入增加约 10%。
  - 这次不接：现金流不变。
- `freelance_referral` 朋友介绍兼职单：
  - 接完整单：本月收入增加约 26%，同时本月支出增加 1800，代表沟通、交通和交付成本。
  - 只接一部分：本月收入增加约 12%。
  - 婉拒这次：现金流不变。

贷款系统暂未加入当前版本。涉及提前还贷、减少利息支出和角色负债字段的设计，后续作为单独系统处理。

## Umami 访问和行为追踪

当前项目已预留 Umami 统计接入，用于长期链接发布后观察访问量和关键行为。

### 如何启用

在 [index.html](./index.html) 中找到：

```html
window.CASH_GAME_UMAMI_WEBSITE_ID = "";
```

把空字符串替换为 Umami 后台提供的 `data-website-id`：

```html
window.CASH_GAME_UMAMI_WEBSITE_ID = "你的 Umami Website ID";
```

留空时不会加载 Umami，也不会发送任何统计数据。

### 项目区分方式

如果多个 GitHub Pages 项目共用同一个域名，例如：

```text
dutongyu413-rgb.github.io
```

本项目会给所有自定义事件自动加上：

```js
project: "cash_game"
```

同时事件名统一使用 `cash_game_` 前缀，便于和其他项目区分。

## 内部测试模式

测试模式只在 URL 带 `debug=1` 或 `seed` 时启用，不影响普通玩家。

```text
http://127.0.0.1:4173/?debug=1&seed=round1
```

可选参数：

- `seed`：固定随机序列；同一种子和同样操作可复现相同结果。
- `event`：强制下一张事件卡，例如 `event=sports_injury`。
- `months`：预设挑战长度，可选 12、24、36。
- `month`：从指定月份开始，仅用于快速验证最终月和后续事件。

调试模式下还可在浏览器控制台使用 `CashGameDebug.forceNextEvent()`、`setSeed()`、`setMonth()`、`setSavings()` 和 `getState()`。

开始一局后，地图右上角会出现 `TEST` 按钮。可视化测试面板支持：

- 修改随机种子、当前月份和现金储备。
- 从全部事件卡中指定下一张测试卡。
- 重置首次新手引导。
- 复制本局诊断信息；自定义身份的具体金额会自动隐藏。

测试面板不会出现在普通试玩链接中，也不会改变正式卡池的随机规则。正式游戏中，同一局不会重复抽到同一张主事件卡。完整的调试路线只保留在内部文档。

### 隐私边界

统计事件不会上传这些具体数据：

- 月收入
- 月支出
- 现金储备
- 安全垫具体数值
- 具体生存分数
- 自定义身份中输入的金额

会上传的是非敏感分档和行为信息，例如：

- 项目标识：`cash_game`
- 挑战长度：12、24 或 36
- 事件类型：健康、收入、选择等
- 是否发生卖出
- 首次开局到第一次掷骰所用的秒数
- 安全垫区间，例如 `1_to_3_months`

## 当前定投和卖出逻辑

当前版本暂时没有完整的基金净值线，但已经在原有定投机制上补了一条轻量后续线。

### 触发流程

1. 玩家可能在前 10 个月随机抽到“低估区域的定投机会”，系统不会在短局结束前强制发出这张卡。
2. 选择“开始定投”后，每月增加 2000 元定投支出。
3. 开始定投约 8-14 个月后，局内时间足够时才会触发“估值修复”。
4. 估值修复约 10-18 个月后，如果玩家仍有持仓且游戏仍在继续，才可能触发“阶段高估”。
5. 玩家可以在后续事件中卖出一半或全部卖出。

投资节点按照 36 个月的自然节奏安排，触发月份不会被压缩到当前挑战结束前。因此 12 个月挑战可能完全遇不到定投机会，也可能在低估阶段结束；24 个月挑战也不保证出现阶段高估。

### 卡牌文案原则

投资后续卡牌只描述市场状态，不替玩家判断账户状态，也不写引导性分析。

当前保留的市场文案：

- `估值修复`：市场从低估区域回到正常估值附近。
- `阶段高估`：市场进入阶段性高估区域。

不要在卡牌里写这些内容：

- “你的账户已经接近回本。”
- “你的账户出现明显浮盈。”
- “现在卖出可以……”
- “继续持有可能……”

玩家自己的状态用数字面板展示：

- 累计投入
- 当前估算市值
- 阶段收益率

### 选择项逻辑

如果定投仍在进行：

- `继续定投`：保持每月投入和当前持仓。
- `停止定投`：停止后续投入，保留已有持仓。
- `卖出一半`：卖出约 50% 持仓，回收现金。
- `全部卖出`：卖出全部持仓，回收现金。

如果定投已经暂停：

- `继续持有`：保持当前持仓。

### 卖出计算

卖出一半和全部卖出使用同一套计算逻辑。

```text
卖出本金 = 当前持仓本金 × 卖出比例
卖出金额 = 卖出本金 × (1 + 当前阶段收益率)
现金储备 += 卖出金额
当前持仓本金 -= 卖出本金
累计卖出本金 += 卖出本金
累计卖出金额 += 卖出金额
已实现盈利/亏损 += 卖出金额 - 卖出本金
```

只要玩家选择卖出，后续定投就会停止：

- 卖出一半：状态变为已暂停，剩余持仓继续保留。
- 全部卖出：状态变为已卖出，剩余持仓归零。

### 最终结算展示

最终结算页需要区分“已经到手的钱”和“还没卖出的估算”。

如果发生过卖出：

- 显示已实现盈利或已实现亏损。
- 计算方式：`卖出金额 - 卖出本金`。

如果还有剩余持仓：

- 显示期末持仓估算。
- 显示未实现盈利或未实现亏损。
- 计算方式：`期末持仓估算 - 剩余持仓本金`。

如果全部卖出：

- 只显示已实现盈利或已实现亏损。
- 不显示未实现部分。

## 累计冲击口径

结果页“最重一击”中的金额标签为“累计冲击”。它只统计本局实际发生过的影响，不按最坏情况或未来可能情况预估。

当前累计方式：

- 一次性现金储备损失：按实际扣掉的金额计入。
- 本月临时收入减少：按本月实际少收入金额计入。
- 本月临时支出增加：按本月实际多支出金额计入。
- 持续性收入减少或支出增加：每结算 1 个月，才累计 1 个月。
- 基础保障生效后：按保障抵扣后的实际储备损失计入。
- 隐藏后续事件：触发当下再计入。

例如“公司业务调整”是收入下降 30%、恢复时间不确定。结果页不会再按最长 12 个月估算；如果实际影响了 4 个月，就只累计 4 个月的收入减少。

## 试玩重点

建议同事测试时重点看这些问题：

- 身份卡的职业、收入、支出、现金储备是否符合直觉。
- 卡牌金额是否贴近生活。
- 同类卡牌是否出现过于频繁。
- 按比例变化的事件是否比固定金额更合理。
- 保障生效时，玩家是否能明显感知到“少花了钱”。
- 定投后续的估值修复、阶段高估和卖出选择是否容易理解；卖出后现金储备、已实现盈利和未实现盈利是否正确展示。
- 地图移动、抽卡、本月结算是否有游戏感。
- 最终结算是否像一局游戏的战报，而不是财务报表。
- 手机窄屏下是否有文字拥挤、按钮换行、内容重叠。

## 手动测试流程

建议至少跑这几类局：

1. 随机身份，完整玩到 12 个月。
2. 选择 36 个月，主动结束游戏，查看最终结算。
3. 配置基础保障后，尽量触发健康风险卡，观察保障提示和结算。
4. 开启定投后玩到结束，观察估值修复、阶段高估、卖出选择和最终定投阶段结论。
5. 分别测试“卖出一半”和“全部卖出”，确认现金储备增加、后续定投停止，最终页能区分已实现盈利和未实现盈利。
6. 选择“离公司近一点”，确认 3 个月后收入提升是否生效。
7. 用支出压力较高的身份测试现金储备被快速消耗的压力局。
8. 用图书管理员、教师等低收入或稳态收入身份测试低收入局。

## 已知待优化方向

- 下一阶段先用6至10名测试者验证反馈入口、评分理解、生活体验和重玩入口是否清楚。

- 卡牌后续优先减少机制相似和标准答案感，不以继续扩充数量为目标。
- 贷款系统与投资净值线属于后置大功能，当前内部测试阶段暂缓。
